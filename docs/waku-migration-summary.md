# Waku移行コスト - エグゼクティブサマリー

## TL;DR

**結論**: Next.jsからWakuへの移行は**推奨しません**

- **移行コスト**: 1-2ヶ月（20-43人日）
- **主なリスク**: Alpha版の不安定性、機能損失、メンテナンスコスト増加
- **推奨**: 現在のNext.js構成を維持し、Wakuは別プロジェクトで学習

## コスト見積もり（要約）

| カテゴリ | 工数 | リスク |
|---------|------|--------|
| ルーティング再構築 | 7-11日 | 高 |
| コンポーネント移行 | 3-4日 | 中 |
| 機能再実装 | 3-4日 | 高 |
| ビルド・デプロイ | 2-3日 | 中 |
| テスト・QA | 3-5日 | 中 |
| **合計** | **20-30日** | **高** |
| **リスク込み** | **27-43日** | - |

## 失われる主要機能

1. **パラレルルート** (@search, @detail) → 手動実装が必要
2. **ファイルベースルーティング** → 全ルートを手動定義
3. **error.tsx/loading.tsx** → React機能で代替
4. **Metadata API** → 手動でheadタグ管理
5. **next/link最適化** → プリフェッチを手動実装
6. **API Routes** → 別サーバーに分離が必要

## Wakuの主な制約

- **本番環境**: Alpha版（安定性・セキュリティ未保証）
- **エコシステム**: プラグイン・ツールが少ない
- **エンタープライズ機能**: ISR、i18n、画像最適化なし
- **ドキュメント**: 不足、試行錯誤が必要

## 推奨アクション

### ✅ やるべきこと
1. **現状維持**: Next.js 16で継続開発
2. **学習**: 別プロジェクトでWakuを試す（1-2週間）
3. **最適化**: Next.jsの既存機能で改善
   - Server Components活用拡大
   - Dynamic Imports
   - バンドル最適化

### ❌ やらないこと
1. 本番アプリのWaku移行（現時点では）
2. Wakuを使った新規プロジェクト開始（学習以外）

### 🔄 再評価のタイミング
- Waku v1.0リリース後（2026年後半予想）
- 本番環境実績の蓄積を確認
- エコシステムの成熟を待つ

## 比較表

| 観点 | Next.js | Waku |
